### チーム開発について

#### チーム開発が破綻しないために

チームでプロダクトを作る場合、それぞれが好き勝手に開発を進めるとプロジェクトが破綻する。一般的に以下のようなプロセスを踏む:

1. 最終形の認識合わせ（要件定義）
2. タスクの洗い出し + 役割分担 + スケジュール決め
3. 実際に開発をしていく（随時進捗を管理）
4. スケジュールやタスクに問題がないか随時チェック

---

### 開発手法

#### ウォーターフォール開発

上流工程から下流工程へと基本的に手戻りなく進んでいく開発モデル。V字モデルで表現されるように、各上流フェーズに対応した検証フェーズが存在する。

| | メリット | デメリット |
|--|---------|----------|
| | 最初に要件をすべて定義するのでスケジュールが組みやすい | 途中で要件や仕様の変更がしにくい |
| | 必要な人的リソースが予め予測しやすい | 検証までのタイムラグが大きくプロジェクトのリスクになる |

#### アジャイル開発

システムは途中で変更されることを前提にした開発手法。「計画」→「設計」→「実装」→「テスト」というサイクルを繰り返しながら機能のリリースを行う。

| | メリット | デメリット |
|--|---------|----------|
| | 初期リリースまでの期間が短く早期にユーザーの反応を得られる | 機能ごとにリリースするため全体のスケジュールを把握しにくい |
| | システムを変更しやすくユーザの要望に応えやすい | 当初の方向性からズレやすく全体整合性が損なわれる場合がある |

#### アジャイル開発の代表的な手法

- **スクラム**: スプリントと呼ばれる短い開発サイクルを繰り返す手法
- **XP（エクストリーム・プログラミング）**: ペアプログラミングやTDDなどのプラクティスを重視する手法
- **カンバン**: タスクの可視化と作業中の数の制限によりフローを管理する手法

---

### MVP（Minimum Viable Product）

早期にユーザのニーズを検証するための、実用最小限のプロダクト。

#### メリット

- 早期にユーザのフィードバックを得るので、無駄な機能を開発するコストを抑えられる
- プロダクトの市場投入を速やかに行える

#### デメリット

- フィードバック結果によっては完全に作り直すようなコストが発生する
- 機能不足の状態でユーザに利用してもらうため、複合的な機能によって成り立つプロダクトの場合は正しいフィードバックが得られないリスクがある

---

### 課題（Must）

1. GitHubのリポジトリを新規に作成する
2. 作ったリポジトリをローカルにクローンする
3. クローンしたリポジトリから別のブランチを作成する（featureブランチ）
4. コーディング規約に沿ったコードに修正する
   - [MDNのコーディング規約](https://developer.mozilla.org/en-US/docs/MDN/Guidelines/Code_guidelines/JavaScript) を参考に下記のコードを修正:

   ```jsx
   function plus(n1,n2){return n1+n2;}

   function f3(a, b){if(a==="test"&&b!=="temp")
   {return b} else{return " " + b + "random text"}}

   var totalnumberOf_catIn_tokyo_atLastYear = 0;
   ```

5. ESLintを導入して下記のルールを追加後、コードを修正する
   - alert関数は使用禁止
   - 変数名は12文字以内に収める
6. ローカルのブランチの変更内容をGitHubに反映する
7. 反映されたブランチからプルリクエストを出してマージする

## Section 6-2〜6-5: ポートフォリオサイト作成（個人プロジェクト）

### 課題

管理画面付きのポートフォリオサイトを作成する。

### 参考URL

- [Firebase ドキュメント](https://firebase.google.com/docs?hl=ja)
- [Firebase Authentication](https://firebase.google.com/docs/auth?hl=ja)
- [環境変数でFirebase configを管理する方法](https://zenn.dev/nino_cast/books/43c539eb47caab/viewer/285689)

---

### 課題（Must）

#### 開発環境

- **フロントエンド**:
  - 言語: TypeScript
  - フレームワーク: Next.js（App router）/ Nuxt.js / Remix から選択
- **バックエンド**:
  - 言語: TypeScript
  - フレームワーク: Express.js / NestJS から選択
  - ORM: Prisma / TypeORM から選択
  - DB: MySQL / PostgreSQL から選択
- **その他**:
  - バックエンドアプリにはフロントエンドアプリのサーバ機能からアクセスすること

#### 機能要件

**ポートフォリオ:**
- トップページ
  - プロフィール（画像付き）
  - 実績
  - スキル
- コンタクト
  - 問い合わせフォーム

**管理画面:**
- トップページのコンテンツを管理できるページ
- 管理者としてログインしたユーザのみアクセス可能

**認証・認可:**
- Firebaseを使って認証・認可機能を実装する（個人のアカウントを利用）
- 認証されていないユーザが管理画面のURLへアクセスした際、HTTPステータスコード404で応答すること（参考: GitHub/Notionの挙動）
- Firebaseのconfigは環境変数で管理すること

**画像データ:**
- 画像データも登録・更新ができるようにする
- 画像データはDBで保持すること（AWSを使用する場合S3でも可）

### 課題（Advanced）

- Redisを利用してキャッシュ機能を実装する
- 本番環境としてAWS上に構築する
- AWSでドメインを取得してアクセスできるようにする（httpsでアクセス可能にする）
